# ADR-0032: Canonical Payload Hashing

## Status

Accepted for Phase 1 with documented limitations.

## Context

Approval must bind to exact tool arguments. Object key ordering should not invalidate logically identical JSON payloads, but payload changes must invalidate approval.

## Decision

Phase 1 uses SHA-256 over deterministic canonical JSON generated by recursively sorting object keys and then applying JavaScript `JSON.stringify` semantics.

This is not a complete RFC 8785 / JSON Canonicalization Scheme implementation.

Detailed behavior and limitations are documented in [Approval Binding](APPROVAL_BINDING.md). Edge-case coverage is implemented in `packages/authority-engine/src/approval-bind.test.ts`.

## Consequences

- Phase 1 is deterministic inside the JavaScript runtime.
- Cross-language clients should not independently compute approval hashes unless they exactly match this profile.
- Future work should adopt RFC 8785-compatible canonicalization or define a formal Ananke canonical payload profile.
- Governance claims apply only to calls routed through Ananke.
